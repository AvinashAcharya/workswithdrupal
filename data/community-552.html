Crumbs breadcrumbs

Posted by donquixote on September 4, 2013 at 3:21pm

This is a placeholder / landing page for Crumbs, the breadcrumbs suite.
The real magic is here: http://drupal.org/project/crumbs

A Google search for "drupal breadcrumbs" does not show Crumbs anywhere in the first pages.
The reason for that is very likely that "drupal.org/project/crumbs" does not contain the term "breadcrumb(s)" in the url.
This placeholder page is an attempt to fix this. Crumbs does not have to be top ranked, but people deserve to know about it at least.

If anyone is unhappy with this, please discuss here, #2072785: Crumbs module page SEO

So, what is it all about?

Crumbs determines breadcrumbs for your site.
It does so by finding the parent of the current page, then the parent of the parent, etc, until it arrives at the home page.

Crumbs comes with a bunch of built-in plugins, that can be individually enabled, disabled and prioritized, and some of which provide further configuration options.

Built-in Plugins

- entity reference
- entityreference_prepopulate
- commerce checkout
- taxonomy
- menu
- organic groups
- path alias
- text fields for breadcrumb item titles
- views with contextual filters
- forum
- blog
- comment
- user-defined entity parent patterns per bundle, based on token

I am also working on some facetapi stuff.. this could be awesome.

Domain Url Alias

Posted by artis on September 12, 2013 at 6:18am

This module is designed as an alternative to the Domain Path module. Whereas Domain Path allows the user to enter custom url aliases for each active domain, this module uses the same alias for all active domains through separate db tables.

The advantage of this approach is that a path like /about can exist on each domain and point to separate nodes. For example, domain1.com/about is an alias for node/23, but domain2.com/about is an alias for node/56. This is not possible with Domain Path.

Cookie-Aware Page Cache

Posted by Les Lim on September 15, 2013 at 8:05am

Cookie-Aware Page Cache module extends Drupal core's default database cache to key page cache entries by cookie in addition to URL.

You may need this if you are preparing different markup for pages depending on the value of a user's browser cookie, and the default database cache is the only caching mechanism available to you. For example, you may be displaying local information on a user's favorite store location, which is kept in a cookie. Or you might be using Context Breakpoint module, which uses a cookie to pass information about the user's screen size to use as a condition in the Context system.

You can specify more than one cookie to take into account when querying the page cache. However, you should only select cookies with limited sets of possible values. The performance benefits of page caching decrease as the number of possible cookie value combinations increases.
